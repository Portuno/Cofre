# Requirements Document: Web Transformation to Vercel + Supabase

## Introduction

This document specifies the requirements for transforming the Cofre Vault Platform from a Rust backend/CLI application into a modern web application deployable on Vercel with Supabase as the backend database. The transformation maintains all existing functionality (audio processing, embeddings, RAG chat, vault management, semantic graph) while introducing a web-based frontend and serverless architecture.

The platform enables users to create collaborative vaults, store and organize content (audio, images, links), generate embeddings for semantic search, and interact with a RAG-powered chat interface that understands vault content through a semantic graph.

## Glossary

- **Vault**: A collaborative container for organizing and sharing content among team members
- **Content_Item**: A discrete piece of content (audio, image, or link) stored within a vault
- **Tag**: A semantic label applied to content items for organization and discovery
- **Semantic_Graph**: An in-memory graph structure representing relationships between content items based on shared tags and embeddings
- **Embedding**: A vector representation of content generated by Gemini API for semantic similarity calculations
- **RAG_Chat**: Retrieval-Augmented Generation chat interface that answers questions using vault content
- **Vault_Member**: A user with a specific role (Owner or Member) in a vault
- **Session_Token**: Authentication token issued after successful login, used for API requests
- **Supabase**: PostgreSQL database service with built-in authentication and pgvector extension
- **Vercel**: Serverless deployment platform for hosting the backend API
- **ElevenLabs**: Third-party service for audio transcription
- **Gemini_API**: Google's AI service for embeddings and LLM-based chat responses
- **pgvector**: PostgreSQL extension for storing and querying vector embeddings
- **Frontend**: Web UI for user interaction (React/Next.js or similar)
- **Backend_API**: Serverless functions handling business logic and database operations
- **Environment_Variables**: Configuration values injected at deployment time (API keys, database URLs, etc.)

## Requirements

### Requirement 1: Web Application Architecture

**User Story:** As a platform architect, I want to transform the Rust backend into a web-deployable architecture, so that users can access the platform through a web browser without installation.

#### Acceptance Criteria

1. THE Web_Application SHALL consist of two independent components: Frontend and Backend_API
2. THE Frontend SHALL be a single-page application (SPA) or server-side rendered application deployable to Vercel
3. THE Backend_API SHALL be implemented as serverless functions deployable to Vercel
4. WHEN a user accesses the application, THE Frontend SHALL communicate with Backend_API exclusively through REST or GraphQL endpoints
5. THE Backend_API SHALL maintain all existing business logic from the Rust services (audio, embeddings, RAG chat, vault, graph, auth, content, tags)
6. WHERE the original Rust application used in-memory Graph structures, THE Backend_API SHALL reconstruct graphs on-demand from database queries
7. THE Backend_API SHALL be stateless to support horizontal scaling on Vercel

### Requirement 2: Supabase Database Integration

**User Story:** As a DevOps engineer, I want to migrate the PostgreSQL database to Supabase, so that the application has a managed, scalable database with built-in authentication.

#### Acceptance Criteria

1. THE Database_Schema SHALL be migrated from the existing PostgreSQL setup to Supabase PostgreSQL
2. THE pgvector Extension SHALL be enabled in Supabase for storing and querying embeddings
3. WHEN the application starts, THE Backend_API SHALL connect to Supabase using the DATABASE_URL environment variable
4. THE Database_Connection SHALL use connection pooling to optimize resource usage on serverless functions
5. ALL existing tables (vaults, vault_members, vault_invites, content_items, tags, item_tags, embeddings) SHALL be created via SQL migrations
6. THE Migrations SHALL be version-controlled and executable in sequence
7. WHERE Supabase provides built-in authentication, THE Backend_API MAY use Supabase Auth for user management
8. THE Database_Schema SHALL support the existing data models without modification to business logic

### Requirement 3: Vercel Deployment Configuration

**User Story:** As a DevOps engineer, I want to configure the application for Vercel deployment, so that the backend runs reliably in a serverless environment.

#### Acceptance Criteria

1. THE Backend_API SHALL be structured as Vercel serverless functions (Node.js, Python, or Rust runtime)
2. WHEN the application is deployed to Vercel, THE Environment_Variables SHALL be injected from Vercel's environment configuration
3. THE Backend_API SHALL have a vercel.json configuration file specifying build and runtime settings
4. THE Backend_API SHALL implement graceful startup and shutdown to handle Vercel's function lifecycle
5. WHEN a request is received, THE Backend_API SHALL respond within Vercel's timeout limits (default 60 seconds for Pro plan)
6. THE Frontend AND Backend_API SHALL be deployable as separate Vercel projects or as a monorepo with separate deployment configurations
7. THE Deployment_Process SHALL include automated database migrations on each deployment
8. WHERE the original application used long-running processes, THE Backend_API SHALL refactor to request-response patterns suitable for serverless

### Requirement 4: Authentication and Authorization

**User Story:** As a security engineer, I want to implement secure authentication and authorization, so that only authorized users can access vaults and content.

#### Acceptance Criteria

1. WHEN a user signs up, THE Auth_Service SHALL create a user account in Supabase Auth
2. WHEN a user signs in, THE Auth_Service SHALL validate credentials and issue a Session_Token
3. THE Session_Token SHALL be a JWT token signed by Supabase Auth
4. WHEN a user makes an API request, THE Backend_API SHALL validate the Session_Token before processing
5. IF the Session_Token is invalid or expired, THEN THE Backend_API SHALL return a 401 Unauthorized response
6. THE Authorization_System SHALL enforce role-based access control (Owner, Member) for vault operations
7. WHEN a user attempts to access a vault, THE Backend_API SHALL verify the user is a Vault_Member with appropriate permissions
8. IF a user lacks permissions, THEN THE Backend_API SHALL return a 403 Forbidden response
9. THE Frontend SHALL store the Session_Token securely (HttpOnly cookie or secure localStorage)
10. WHERE Supabase Auth is used, THE Backend_API SHALL leverage Supabase's built-in JWT validation

### Requirement 5: API Endpoints for Frontend Integration

**User Story:** As a frontend developer, I want well-defined REST API endpoints, so that I can build a responsive web interface.

#### Acceptance Criteria

1. THE Backend_API SHALL expose REST endpoints for all vault operations (create, read, update, delete, list)
2. THE Backend_API SHALL expose REST endpoints for all content operations (upload, retrieve, tag, search)
3. THE Backend_API SHALL expose REST endpoints for all tag operations (create, update, delete, list)
4. THE Backend_API SHALL expose a REST endpoint for RAG chat (POST /api/chat with message payload)
5. THE Backend_API SHALL expose a REST endpoint for semantic graph queries (GET /api/graph/:vault_id)
6. THE Backend_API SHALL expose authentication endpoints (POST /api/auth/signup, POST /api/auth/signin, POST /api/auth/signout)
7. WHEN a request is received, THE Backend_API SHALL return responses in JSON format with appropriate HTTP status codes
8. THE Backend_API SHALL implement pagination for list endpoints (limit, offset parameters)
9. THE Backend_API SHALL implement filtering and sorting for content queries
10. WHERE the original Rust application used custom protocols, THE Backend_API SHALL translate to standard REST conventions

### Requirement 6: Audio Processing and Transcription

**User Story:** As a user, I want to upload audio files and have them automatically transcribed, so that I can search and reference audio content by text.

#### Acceptance Criteria

1. WHEN a user uploads an audio file, THE Backend_API SHALL accept multipart/form-data requests
2. THE Backend_API SHALL store the audio file in Supabase Storage or an external storage service
3. WHEN an audio file is uploaded, THE Backend_API SHALL send it to ElevenLabs for transcription
4. WHEN transcription completes, THE Backend_API SHALL store the transcript in the database
5. THE Backend_API SHALL generate an embedding for the transcript using Gemini API
6. THE Backend_API SHALL store the embedding in the database using pgvector
7. IF transcription fails, THEN THE Backend_API SHALL log the error and return a descriptive error message
8. THE Audio_Processing SHALL be asynchronous to avoid blocking the user's request
9. WHERE the original application used in-memory processing, THE Backend_API SHALL use database-backed queues or Vercel's background functions

### Requirement 7: Embedding Generation and Storage

**User Story:** As a system, I want to generate and store embeddings for all content, so that semantic search and RAG chat can function.

#### Acceptance Criteria

1. WHEN content is created or updated, THE Backend_API SHALL generate an embedding using Gemini API
2. THE Embedding_Model SHALL be configurable via the EMBEDDING_MODEL environment variable
3. THE Backend_API SHALL store embeddings in the database using pgvector
4. WHEN querying for similar content, THE Backend_API SHALL use pgvector's similarity search
5. THE Similarity_Threshold SHALL be configurable via the SIMILARITY_THRESHOLD environment variable
6. IF embedding generation fails, THEN THE Backend_API SHALL retry with exponential backoff
7. THE Backend_API SHALL cache embeddings to avoid regenerating for unchanged content
8. WHERE the original application used in-memory embeddings, THE Backend_API SHALL persist embeddings in the database

### Requirement 8: Semantic Graph Construction and Querying

**User Story:** As a user, I want the system to understand relationships between content items, so that I can discover related content and get contextual chat responses.

#### Acceptance Criteria

1. WHEN a user queries the semantic graph, THE Backend_API SHALL reconstruct the graph from database queries
2. THE Graph_Construction SHALL identify edges between content items based on shared tags and embedding similarity
3. WHEN two content items share a tag, THE Backend_API SHALL create an edge with weight based on embedding similarity
4. THE Backend_API SHALL support filtering the graph by tag or content type
5. WHEN a user requests the graph, THE Backend_API SHALL return a JSON representation with nodes and edges
6. THE Graph_Query SHALL complete within 2 seconds for vaults with up to 10,000 content items
7. WHERE the original application used in-memory graphs, THE Backend_API SHALL compute graphs on-demand from database data
8. THE Backend_API SHALL implement caching for frequently accessed graphs

### Requirement 9: RAG Chat Interface

**User Story:** As a user, I want to ask questions about vault content and receive contextual answers, so that I can quickly find information without manual searching.

#### Acceptance Criteria

1. WHEN a user sends a chat message, THE Backend_API SHALL retrieve relevant content from the vault using semantic search
2. THE Backend_API SHALL use the Semantic_Graph to identify related content items
3. THE Backend_API SHALL construct a context window from the top-K most relevant content items
4. WHEN generating a response, THE Backend_API SHALL use Gemini API with the context window
5. THE LLM_Model SHALL be configurable via the LLM_MODEL environment variable
6. THE Backend_API SHALL include references to source content items in the chat response
7. IF the vault has insufficient content, THEN THE Backend_API SHALL return a response indicating limited context
8. THE Chat_Response SHALL include a list of referenced_node_ids for frontend display
9. WHERE the original application used in-memory context, THE Backend_API SHALL query the database for each chat request

### Requirement 10: Environment Configuration for Vercel

**User Story:** As a DevOps engineer, I want to manage environment variables securely, so that sensitive credentials are not exposed in code.

#### Acceptance Criteria

1. THE Backend_API SHALL read all configuration from environment variables
2. THE Required_Environment_Variables SHALL include: DATABASE_URL, SUPABASE_URL, SUPABASE_KEY, GEMINI_API_KEY, ELEVENLABS_API_KEY
3. THE Optional_Environment_Variables SHALL include: EMBEDDING_MODEL, LLM_MODEL, SIMILARITY_THRESHOLD, API_ADDR, RUST_LOG
4. WHEN the application starts, THE Backend_API SHALL validate that all required environment variables are set
5. IF a required environment variable is missing, THEN THE Backend_API SHALL fail fast with a descriptive error message
6. THE Environment_Variables SHALL be configured in Vercel's project settings
7. THE .env.example file SHALL document all environment variables with descriptions
8. WHERE environment variables are used in code, THE Backend_API SHALL use a configuration struct to centralize access

### Requirement 11: Database Migrations for Supabase

**User Story:** As a database administrator, I want version-controlled migrations, so that schema changes are reproducible and reversible.

#### Acceptance Criteria

1. THE Migrations SHALL be stored in a migrations/ directory with sequential numbering (001_, 002_, etc.)
2. EACH Migration SHALL include both UP and DOWN scripts for reversibility
3. THE Migrations SHALL create all necessary tables: vaults, vault_members, vault_invites, content_items, tags, item_tags, embeddings
4. THE Migrations SHALL enable the pgvector extension
5. THE Migrations SHALL create appropriate indexes for performance (on vault_id, user_id, tag_id, embedding similarity)
6. WHEN the application starts, THE Backend_API SHALL automatically run pending migrations
7. THE Migration_Tracking SHALL use a migrations table to record which migrations have been applied
8. IF a migration fails, THEN THE Backend_API SHALL log the error and prevent application startup
9. WHERE the original application used raw SQL, THE Migrations SHALL use standard PostgreSQL syntax compatible with Supabase

### Requirement 12: Frontend-Backend Communication

**User Story:** As a frontend developer, I want clear contracts for API communication, so that I can build a reliable user interface.

#### Acceptance Criteria

1. THE Backend_API SHALL return all responses in JSON format
2. THE Backend_API SHALL use standard HTTP status codes (200, 201, 400, 401, 403, 404, 500)
3. WHEN an error occurs, THE Backend_API SHALL return a JSON error response with error_code and message fields
4. THE Backend_API SHALL implement CORS headers to allow requests from the Frontend domain
5. THE Backend_API SHALL support both GET and POST requests for appropriate operations
6. THE Backend_API SHALL validate all input data and return 400 Bad Request for invalid data
7. WHERE the original application used custom serialization, THE Backend_API SHALL use standard JSON serialization
8. THE API_Documentation SHALL be available (OpenAPI/Swagger or similar)

### Requirement 13: Performance and Scalability

**User Story:** As a platform operator, I want the application to handle growing user load, so that performance remains acceptable as the user base grows.

#### Acceptance Criteria

1. THE Backend_API SHALL implement database connection pooling to optimize resource usage
2. THE Backend_API SHALL cache frequently accessed data (vaults, tags, graphs) with appropriate TTL
3. WHEN a user requests a large list of content items, THE Backend_API SHALL implement pagination with default limit of 50 items
4. THE Backend_API SHALL implement query optimization to avoid N+1 queries
5. WHEN embedding generation is requested, THE Backend_API SHALL implement rate limiting to avoid exceeding Gemini API quotas
6. THE Backend_API SHALL implement request timeouts to prevent hanging requests
7. WHERE the original application used in-memory caching, THE Backend_API SHALL use Redis or Supabase caching
8. THE Backend_API SHALL monitor response times and log slow queries

### Requirement 14: Security Best Practices

**User Story:** As a security engineer, I want to implement security best practices, so that user data and system integrity are protected.

#### Acceptance Criteria

1. THE Backend_API SHALL validate all user input to prevent SQL injection
2. THE Backend_API SHALL use parameterized queries for all database operations
3. THE Backend_API SHALL implement rate limiting on authentication endpoints to prevent brute force attacks
4. THE Backend_API SHALL use HTTPS for all communication
5. THE Backend_API SHALL implement CORS to restrict cross-origin requests
6. THE Backend_API SHALL sanitize error messages to avoid information disclosure
7. THE Backend_API SHALL log security events (failed authentication, unauthorized access attempts)
8. WHERE sensitive data is stored, THE Backend_API SHALL use encryption at rest
9. THE Backend_API SHALL implement request validation to prevent oversized payloads

### Requirement 15: Monitoring and Logging

**User Story:** As a platform operator, I want visibility into application behavior, so that I can diagnose issues and optimize performance.

#### Acceptance Criteria

1. THE Backend_API SHALL log all API requests with timestamp, method, path, status code, and response time
2. THE Backend_API SHALL log all errors with stack traces and context
3. THE Backend_API SHALL log database operations (queries, connection pool status)
4. WHEN an error occurs, THE Backend_API SHALL include a request ID for tracing
5. THE Logs SHALL be structured (JSON format) for easy parsing and analysis
6. THE Backend_API SHALL integrate with Vercel's logging system
7. WHERE the original application used console logging, THE Backend_API SHALL use a structured logging library
8. THE Backend_API SHALL expose metrics for monitoring (request count, error rate, response time percentiles)

### Requirement 16: Content Storage and Retrieval

**User Story:** As a user, I want to store various types of content (audio, images, links), so that I can organize diverse information in vaults.

#### Acceptance Criteria

1. THE Backend_API SHALL support storing content metadata in the database
2. WHEN a user uploads audio or image files, THE Backend_API SHALL store files in Supabase Storage or external storage
3. THE Backend_API SHALL generate a URL for accessing stored files
4. WHEN a user provides a link, THE Backend_API SHALL store the URL and optionally fetch metadata
5. THE Backend_API SHALL support retrieving content by ID, vault, or tag
6. THE Backend_API SHALL support updating content metadata (title, tags, transcript)
7. WHEN a user deletes content, THE Backend_API SHALL remove the content and associated embeddings
8. WHERE the original application stored content in memory, THE Backend_API SHALL persist to Supabase Storage

### Requirement 17: Vault Collaboration Features

**User Story:** As a user, I want to collaborate with others on shared vaults, so that teams can organize and access content together.

#### Acceptance Criteria

1. WHEN a user creates a vault, THE Backend_API SHALL assign the creator as the Owner
2. THE Owner SHALL be able to invite other users to the vault
3. WHEN an invitation is sent, THE Backend_API SHALL create a VaultInvite with an expiration time
4. WHEN a user accepts an invitation, THE Backend_API SHALL create a VaultMember record with Member role
5. THE Owner SHALL be able to remove members from the vault
6. THE Owner SHALL be able to change member roles
7. WHEN a member is removed, THE Backend_API SHALL revoke their access to vault content
8. THE Backend_API SHALL enforce role-based permissions for all vault operations

### Requirement 18: Error Handling and Recovery

**User Story:** As a system, I want to handle errors gracefully, so that users receive helpful feedback and the system remains stable.

#### Acceptance Criteria

1. WHEN a database connection fails, THE Backend_API SHALL retry with exponential backoff
2. WHEN an external API call fails (ElevenLabs, Gemini), THE Backend_API SHALL retry with exponential backoff
3. IF a retry fails after maximum attempts, THEN THE Backend_API SHALL return a descriptive error to the user
4. WHEN a request times out, THE Backend_API SHALL return a 504 Gateway Timeout response
5. THE Backend_API SHALL implement circuit breakers for external API calls
6. WHERE the original application used panic!, THE Backend_API SHALL return proper error responses
7. THE Backend_API SHALL log all errors with sufficient context for debugging

### Requirement 19: Testing and Quality Assurance

**User Story:** As a developer, I want comprehensive tests, so that I can confidently deploy changes without introducing regressions.

#### Acceptance Criteria

1. THE Backend_API SHALL have unit tests for all business logic
2. THE Backend_API SHALL have integration tests for API endpoints
3. THE Backend_API SHALL have database tests using a test database
4. WHEN tests are run, THE Test_Suite SHALL complete in under 5 minutes
5. THE Test_Coverage SHALL be at least 80% for critical paths
6. WHERE the original application used manual testing, THE Backend_API SHALL use automated tests
7. THE Tests SHALL be executable in CI/CD pipeline

### Requirement 20: Documentation and Developer Experience

**User Story:** As a developer, I want clear documentation, so that I can understand the system and contribute effectively.

#### Acceptance Criteria

1. THE Backend_API SHALL have a README with setup instructions
2. THE Backend_API SHALL have API documentation (OpenAPI/Swagger)
3. THE Backend_API SHALL have inline code comments for complex logic
4. THE Database_Schema SHALL be documented with table descriptions and relationships
5. THE Environment_Variables SHALL be documented in .env.example
6. THE Deployment_Process SHALL be documented with step-by-step instructions
7. WHERE the original application lacked documentation, THE Backend_API SHALL include comprehensive guides
